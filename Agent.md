# Agent 実装ガイド

保守性・可読性・拡張性を重視して実装する。
クリーンアーキテクチャおよびドメイン駆動設計を参考にする。
ドメインロジックをGUIコードやデータ取得コードに混在させないことを最優先する。

# 基本構造

原則として、以下の責務に分けて実装する。

- ドメイン層
- ユースケース層
- インフラ層
- プレゼンテーション層 / GUI層

層の責務が混ざらない範囲で、シンプルな構成を優先する。

# 依存方向

依存方向は原則として以下を守る。

GUI層 → ユースケース層 → ドメイン層

インフラ層は、外部データ取得・保存・変換の詳細を担当する。
ドメイン層は、GUI層・インフラ層・外部API・ファイル形式・DB構造に依存しない。

禁止事項：

- GUIコード内に業務判定ロジックを書くこと
- データ取得コード内に業務判定ロジックを書くこと
- ドメイン層からGUI部品を参照すること
- ドメイン層からCSV、Excel、DB、APIなどの具体的な取得手段を参照すること
- 外部データのDTOをそのままドメインロジックの中心にすること

# ドメイン層

ドメイン層は、業務ルール、判定ロジック、計算ロジックを表現する。

主に以下を置く。

- Entity
- Value Object
- Domain Service
- 純粋な計算関数
- 判定ロジック

Entityは、業務上意味のある対象を表す。
Value Objectは、値そのものに意味があるものを表す。
Domain Serviceは、特定のEntityやValue Objectだけに自然に所属しない業務ロジックを扱う。
ドメイン層は、入力値が同じなら同じ結果を返すように、できるだけ副作用の少ない実装にする。
ファイル入出力、GUI更新、API通信、DBアクセスは行わない。

# ユースケース層

ユースケース層は、アプリケーション固有の処理手順を担当する。

主に以下を行う。

- GUI層からの要求を受ける
- 必要なデータを取得する
- ドメイン層のロジックを呼び出す
- 複数の処理を順序立てて実行する
- 結果をプレゼンテーション層へ返す

ユースケース層は処理のオーケストレーションを担当する。
ただし、GUI表示の詳細や、CSV/API/DBなどの具体的な取得処理は持たない。

# インフラ層

インフラ層は、外部との入出力を担当する。

例：

- CSV読み込み
- API通信
- HTMLパース
- ファイル保存
- 設定ファイル読み込み
- ログ出力

外部データは必要に応じてDTOとして受け取る。
DTOは外部データ形式を表すためのものであり、ドメインロジックの中心にはしない。

必要に応じて、DTOをMapperでDomain Modelへ変換する。

基本方針：

外部データ → DTO → Mapper → Domain Model → UseCase / Domain Logic

# リポジトリ

リポジトリを使う場合、リポジトリは「層」ではなく、データ取得を抽象化するための境界として扱う。

開発規模が大きくなり、テスト容易性やデータ取得元の差し替えが必要になった場合に、Repository Interface / Port を導入する。

導入する場合は以下の方針とする。

- Repository Interface / Port はユースケース層またはドメインに近い側に置く
- 具体的なRepository実装はインフラ層に置く
- UseCaseは具体実装ではなくInterfaceに依存する
- 実装の注入はアプリケーションのエントリポイントで行う

ただし、Repository Interface、Port、依存性注入、DI Containerの導入は、必要性を確認してから行う。

# プレゼンテーション層 / GUI層

GUI層は、ユーザー操作を受け取り、ユースケースを呼び出す。

主に以下を行う。

- ボタン操作、入力値変更などのイベント処理
- 入力値の取得
- ユースケースの呼び出し
- ユースケース結果の表示
- 表示更新
- エラーメッセージ表示

GUI層には、業務判定ロジックや計算ロジックを置かない。

ユースケースから返された結果は、必要に応じてViewModelまたは表示用データに整形して表示する。
表示文言、色、アイコン、表形式などの見た目に関する処理はプレゼンテーション層で扱う。

# ViewModel / 表示用データ

ドメインモデルをそのままGUIに渡してもよいが、表示要件が増えてきた場合はViewModelを導入する。
ViewModelは、GUI表示に都合のよい形に整形されたデータである。

例：
- 表示名
- 単位付き文字列
- 小数点丸め済みの値
- 警告メッセージ
- 表示色
- 表示可否
- 並び順
- セクション名

ただし、ViewModelに業務判定の本体を置かない。
業務判定はドメイン層またはユースケース層で行う。

# 抽象化の導入基準

以下の条件が出てきた場合に、Port、Repository Interface、Mapper、DIを検討する。

- データ取得元を差し替える可能性がある
- CSVとAPIなど、複数の入力元を扱う
- テスト時に外部データ取得をモック化したい
- GUIなしでロジックだけをテストしたい
- 同じドメインロジックを複数画面や複数スクリプトで使う
- ファイル数や機能数が増え、責務の混在が目立ってきた

これらがない段階では、過剰なInterface化やDI Container導入は避ける。

# 実装上の優先順位

優先順位は以下とする。

1. ドメインロジックをGUIやデータ取得処理から分離する
2. 読みやすい命名にする
3. 関数・クラスの責務を小さく保つ
4. テストしやすい純粋関数を増やす
5. 必要になるまで抽象化しすぎない
6. 変更頻度の高い部分と低い部分を分ける
7. 外部データ形式の変更がドメイン層に波及しないようにする

# 判断に迷った場合

判断に迷った場合は、以下を優先する。

- まず動くこと
- ただしドメインロジックをGUIに書かないこと
- 外部データ形式をドメイン層に漏らさないこと
- 将来の拡張より、現在の読みやすさを優先すること
- 抽象化は必要になってから導入すること

## 条件判定

- 単一の数値を閾値帯ごとのランク、点数、ラベルへ変換する場合は、型付きの `RangeTable` を優先して使う。各帯の境界（`>` / `>=`）、既定値、並び順を明示し、判定結果が既存仕様と一致する回帰テストを追加する。
- 複数の独立した条件を合算する採点、または「なぜ成立したか」を後で表示・監査する必要がある複合判定では、`SignalAtom` の採用を提案する。Atom には安定した識別子、成立可否、加点、必要ならグループ加点上限を持たせる。
- 優先順位つきの分岐、例外、他指標に依存する補正は、RangeTable や SignalAtom に無理に詰め込まない。Domain Policy に判定順と例外理由を明示する。
- 既存判定を移行する際は、閾値ちょうど・直上/直下・欠損値・例外値の回帰テストを先に揃え、利用者向けの出力を変えない。
